# 买入否决条件 V1 设计

## 背景

当前系统已经具备缠论结构、买卖点、中枢摘要、市场环境、技术辅助和回测验证能力。现有逻辑会对买入信号加分或减分，但缺少明确的“不要买”规则：当结构出现明显风险时，即使存在买点候选，也应该先否决买入，改为观察。

本阶段目标是增加“买入否决条件”，减少在结构破坏、技术走弱或环境逆风时误触发买入。

## 目标

1. 新增独立的 `VetoContext`，说明是否否决买入、否决强度、触发原因和白话摘要。
2. 买入信号触发否决后，外部动作改为“继续持有”，内部信号改为“观察”。
3. 保留原买点信息，方便用户知道“原本为什么想买，又为什么现在不买”。
4. Web、CLI JSON、CSV 结果都能看到否决信息。
5. 回测自然使用否决后的信号，便于后续验证否决条件是否减少坏信号。

## 非目标

本阶段只否决买入类信号，不否决卖出或减仓信号。卖出信号属于风险控制，应保持敏感。本阶段也不做可配置规则引擎，不引入外部参数管理，不接实盘交易。

## 否决规则

### 硬否决

硬否决会直接把买入类信号改为观察：

1. 三买候选后，最新价格重新跌回最近中枢内部或中枢下方。
2. 技术辅助为“压缩后向下突破”。
3. 日线结构仍为下跌趋势，且买点不是“一买”或“二买”。
4. 买点失效价已经被最新价格触发。

### 组合否决

组合否决用于处理单项风险不够强，但叠加后不适合买入的场景：

1. 市场环境为“逆风”且技术辅助为“拖累”。
2. 30 分钟确认缺失且技术辅助为“拖累”。
3. 最新价格仍在中枢内部，且当前没有明确买点。

组合否决触发后也把买入类信号改为观察，但摘要会说明这是多项条件叠加，而不是单一结构失效。

## 信号处理方式

`ChanSignalEngine.evaluate()` 先按现有逻辑生成原始信号，再调用否决规则。如果原始信号不是买入动作，否决规则只记录可读信息，不改变信号。

当买入被否决：

- `signal` 改为 `观察`。
- `action` 改为 `继续持有`。
- `confidence` 降到不高于 `0.48`。
- `risk_notes` 增加否决摘要。
- `reasons` 保留原买点原因，并追加“买入被否决”的说明。
- `trade_point` 保留，便于用户理解原始买点。

## 数据模型

新增 `VetoContext`：

- `vetoed`: 是否否决买入。
- `level`: `hard`、`combined` 或 `none`。
- `summary`: 面向用户的一句话说明。
- `reasons`: 触发的否决原因。
- `original_signal`: 被否决前的内部信号。
- `original_action`: 被否决前的外部动作。

`ChanSignal` 增加 `veto_context: VetoContext | None`。

## 展示方式

Web 交易信号栏新增“否决条件”：

- 未触发：显示“未触发”。
- 已触发：显示“已否决买入”，并用旁边说明解释原因。

CSV 列表页增加一列“否决条件”，用于快速扫出哪些股票虽然有买点但暂时不适合买。

JSON 输出完整包含 `veto_context`，方便后续回测或外部系统分析。

## 测试计划

1. 单元测试硬否决：
   - 三买后价格跌回中枢，买入改观察。
   - 技术辅助为压缩后向下突破，买入改观察。
2. 单元测试组合否决：
   - 市场逆风叠加技术拖累，买入改观察。
   - 30 分钟确认缺失叠加技术拖累，买入改观察。
3. 单元测试未否决：
   - 一买或二买结构修复不被下跌趋势直接否决。
   - 非买入信号不被否决规则改写。
4. Web API JSON 测试确认 `veto_context` 可序列化。
5. 全量测试和 CLI 冒烟确认旧流程不破坏。

## 成功标准

1. 买入信号遇到关键风险时能明确降为观察。
2. 用户能看懂“为什么有买点但暂时不买”。
3. 买卖点、结构摘要、回测入口保持兼容。
4. 测试通过，且本阶段变更能独立回退。
