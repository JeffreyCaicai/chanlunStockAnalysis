# 缠论交易信号 V1 设计

## 目标

在现有 A 股本地分析工具上新增一个缠论交易信号模块。第一版服务波段交易，主级别为日线，确认级别为 30 分钟。系统输出面向交易动作的三类结论：

- `买入`
- `卖出`
- `继续持有`

内部保留更细的信号分层，以便解释信号强弱和后续扩展：

- `强买入`
- `试买入`
- `持有`
- `观察`
- `减仓`
- `清仓卖出`

第一版不是自动交易系统，也不直接下单。它提供结构化信号、触发原因、置信度、失效条件和持仓相关建议，供人工决策使用。

V1 同时提供 CLI 和本地 Web UI：

- CLI 用于批量分析、JSON 输出和自动化调用。
- 本地 Web UI 用于人工操作、查看信号解释、维护临时持仓输入和导出结果。

Web UI 只在本机运行，不包含登录、多用户、云端部署或实盘交易能力。

## 范围

### V1 包含

- 日线主级别分析。
- 30 分钟确认级别分析，在数据源可用时启用。
- HTTP 日线 K 线兜底，保证模块可以运行。
- mootdx 30 分钟 K 线增强，若当前环境可用则启用。
- 无持仓时输出技术信号。
- 有持仓时结合成本和仓位输出实战动作。
- CLI 参数输入持仓信息。
- 本地 CSV 输入持仓信息。
- 本地 Web UI 操作界面。
- Web UI 输入单票代码、成本价和仓位。
- Web UI 导入本地 CSV 持仓文件。
- Web UI 查看信号、原因、失效条件、数据降级状态和 JSON 结果。
- 收盘后正式信号。
- 盘中预警信号，必须标记为未确认。
- 每条信号输出原因、置信度、失效条件和数据降级状态。

### V1 不包含

- 自动下单。
- 组合级资金管理优化。
- 标准完整线段算法。
- 标准完整中枢算法。
- 严格一买、二买、三买、一卖、二卖、三卖全套识别。
- 对短线 B（30 分钟主级别 + 5 分钟确认）的完整支持。
- 对中线 C（周线主级别 + 日线确认）的完整支持。
- 用户登录、权限系统和多用户协作。
- 云端部署、远程数据库和任务队列。
- 浏览器内直接下单或券商交易接口。

短线 B 和中线 C 后续作为扩展模式实现，V1 只保留接口方向，不实现其完整策略规则。

## 产品界面设计

### CLI

CLI 是信号模块的基础入口，必须先保持稳定。它用于：

- 单票临时分析。
- 批量持仓 CSV 分析。
- JSON 输出。
- 后续脚本化、自动化或定时任务调用。

### Local Web UI

Web UI 是本地操作界面，服务人工查看和交互，不替代 CLI。

V1 页面布局：

- 左侧：自选股或持仓列表。
- 中间：当前股票结构状态，包括日线主级别、30 分钟确认状态、分型、笔、趋势状态和降级提示。
- 右侧：交易信号卡片，包括外部动作、内部信号、置信度、触发原因、失效条件、风险提示和 JSON 导出。

V1 页面能力：

- 输入单个股票代码并运行分析。
- 输入或修改持仓成本价。
- 输入或修改仓位比例。
- 导入本地 CSV 持仓文件。
- 点击持仓列表切换股票。
- 查看 `买入`、`卖出`、`继续持有`。
- 查看内部信号：`强买入`、`试买入`、`持有`、`观察`、`减仓`、`清仓卖出`。
- 查看 `confirmation_missing` 和 30 分钟确认状态。
- 查看信号原因、失效条件和风险提示。
- 复制或下载当前 JSON 结果。

V1 Web UI 不包含：

- 在线账户系统。
- 历史信号数据库。
- 图形化完整 K 线绘制。
- 自动刷新后台任务。
- 实盘交易接口。

图形化 K 线可以作为后续版本加入。V1 只需要把结构判断和信号解释展示清楚。

### 本地服务边界

Web UI 使用本地服务承载：

- 前端页面请求本地 HTTP API。
- 本地 API 复用 CLI 相同的分析服务。
- CLI 和 Web UI 必须共享同一套核心信号引擎，避免两套逻辑分叉。
- Web UI 只读本地上传或指定的 CSV 文件，不把数据上传到外部服务。

## 数据源设计

### 双源策略

K 线数据采用双源策略：

- HTTP 日线源：作为基础数据源，优先保证系统可运行。
- mootdx：作为增强数据源，用于获取 30 分钟 K 线确认。

如果 mootdx 不可用，系统不得失败退出；应继续输出日线信号，并在结果中标记：

- `confirmation_missing: true`
- `confidence` 下调
- `reasons` 中说明缺少 30 分钟确认

### K 线模型

统一 K 线数据模型包含：

- `code`
- `period`
- `timestamp`
- `open`
- `high`
- `low`
- `close`
- `volume`
- `amount`

所有缠论规则只依赖统一模型，不直接依赖具体数据源返回格式。

## 缠论结构设计

V1 使用渐进式实用规则，先构建可解释信号，再逐步扩展标准结构。

### K 线包含关系处理

相邻 K 线出现包含关系时，按当前方向进行合并：

- 上行方向：合并后高点取较高值，低点取较高值。
- 下行方向：合并后高点取较低值，低点取较低值。
- 方向不明时，用最近非包含 K 线的高低点变化确认方向。

合并后的 K 线用于后续分型和笔识别。

### 分型识别

顶分型：

- 中间 K 线高点高于左右两根 K 线高点。
- 中间 K 线低点高于左右两根 K 线低点。

底分型：

- 中间 K 线低点低于左右两根 K 线低点。
- 中间 K 线高点低于左右两根 K 线高点。

分型结果包含：

- 类型：`top` 或 `bottom`
- 时间
- 价格
- 对应 K 线索引

### 笔结构近似

V1 使用实用笔近似：

- 相邻有效分型必须顶底交替。
- 顶底分型之间至少间隔一定数量的合并 K 线，默认 4。
- 如果连续出现同类型分型，保留更极端的那个。
- 笔方向为 `up` 或 `down`。

笔结构结果用于判断日线方向、结构破坏和 30 分钟触发。

### 趋势/震荡判断

日线主级别状态分为：

- `uptrend`
- `downtrend`
- `range`
- `unknown`

V1 判断规则：

- 最近上行笔高点和低点逐步抬高，判断为 `uptrend`。
- 最近下行笔高点和低点逐步降低，判断为 `downtrend`。
- 最近笔高低点交错且无明确方向，判断为 `range`。
- 笔数量不足，判断为 `unknown`。

### 背驰近似

V1 不实现严格中枢背驰，先实现力度近似：

- 比较同方向最近两笔的价格幅度。
- 比较同方向最近两笔的成交量。
- 如果价格创新高但幅度或成交量明显收缩，标记上行背驰风险。
- 如果价格创新低但幅度或成交量明显收缩，标记下行背驰修复可能。

背驰近似只作为信号加减分，不单独触发强信号。

## 信号规则

### 内部信号

`强买入`：

- 日线状态不是 `downtrend`。
- 日线最近结构出现底分型后向上笔确认。
- 30 分钟确认可用，并出现回抽不破或向上笔恢复。
- 未触发明显上行背驰风险。

`试买入`：

- 日线出现潜在底分型或下跌力度衰竭。
- 30 分钟确认缺失或只出现早期修复迹象。
- 必须给出明确失效条件，例如跌破最近日线底分型低点。

`持有`：

- 已有持仓。
- 日线结构未破坏。
- 未出现明确减仓或清仓条件。

`观察`：

- 无持仓或信号不充分。
- 结构处于震荡或未知。
- 买卖条件均未满足。

`减仓`：

- 已有持仓。
- 日线结构转弱，或 30 分钟出现向下笔破坏。
- 上行背驰风险较高但尚未触发清仓条件。

`清仓卖出`：

- 日线结构跌破关键底分型或最近上行笔低点。
- 30 分钟确认向下破坏，若确认可用。
- 或结合持仓成本触发风控条件。

### 外部动作映射

- `强买入`、`试买入` 映射为 `买入`。
- `持有`、`观察` 映射为 `继续持有`。
- `减仓`、`清仓卖出` 映射为 `卖出`。

输出必须同时包含内部信号和外部动作，避免只有单一结论而无法解释。

## 持仓输入

### CLI 参数

单票临时分析支持：

- `--cost`
- `--position`

示例：

```bash
PYTHONPATH=src python -m astockdata.chan_cli 688017 --cost 120.5 --position 0.3
```

### CSV 文件

批量持仓分析支持 CSV：

```csv
code,cost,position
688017,120.5,0.3
600519,1500,0.2
```

字段含义：

- `code`：6 位 A 股代码。
- `cost`：持仓成本价。
- `position`：仓位比例，取值 0 到 1。

有持仓输入时，系统结合成本和仓位调整信号解释：

- 盈利但结构转弱：倾向减仓。
- 亏损且跌破结构失效位：倾向清仓卖出。
- 盈利且结构未破坏：倾向继续持有。
- 无持仓但出现试买信号：输出买入，但提示试错属性和失效位。

## 信号时态

### 正式信号

收盘后使用已完成 K 线生成正式信号：

- `intraday: false`
- `confirmed: true`

### 盘中预警

盘中可以生成预警，但不得覆盖正式信号：

- `intraday: true`
- `confirmed: false`
- `warning: true`

盘中预警必须在原因中说明当前 K 线未完成，信号可能变化。

## 输出结构

每条信号输出：

- `code`
- `action`
- `signal`
- `confidence`
- `confirmed`
- `intraday`
- `confirmation_missing`
- `reasons`
- `invalidations`
- `risk_notes`
- `position_context`

示例：

```json
{
  "code": "688017",
  "action": "买入",
  "signal": "试买入",
  "confidence": 0.62,
  "confirmed": true,
  "intraday": false,
  "confirmation_missing": true,
  "reasons": [
    "日线下跌力度收缩，出现潜在底分型",
    "30分钟确认数据不可用，信号降级"
  ],
  "invalidations": [
    "跌破最近日线底分型低点 118.20"
  ],
  "risk_notes": [
    "该信号为试买入，不适合重仓"
  ],
  "position_context": null
}
```

支持：

## Web API 设计

本地 Web UI 暴露最小 API：

`GET /`

- 返回本地 Web 页面。

`POST /api/analyze`

请求：

```json
{
  "code": "688017",
  "cost": 120.5,
  "position": 0.3,
  "intraday": false
}
```

响应为标准信号输出结构。

`POST /api/analyze-portfolio`

请求：

```json
{
  "holdings": [
    {"code": "688017", "cost": 120.5, "position": 0.3},
    {"code": "600519", "cost": 1500.0, "position": 0.2}
  ],
  "intraday": false
}
```

响应：

```json
{
  "results": []
}
```

其中 `results` 每项为标准信号输出结构。

`GET /api/health`

- 返回本地服务状态。
- 包含数据源可用性概要，例如 HTTP 日线源是否可用、mootdx 是否可用。

## Web UI 启动方式

## 风险与边界

- V1 的笔和背驰是实用近似，不声称等同严格缠论全结构。
- 数据源异常时必须明确降级，不得静默给出高置信信号。
- 所有信号都是辅助决策，不构成投资建议。
- 自动交易、实盘下单和资金管理优化不在 V1 范围内。
- Web UI 是本地辅助界面，不提供交易安全、账户安全或部署安全保障。
